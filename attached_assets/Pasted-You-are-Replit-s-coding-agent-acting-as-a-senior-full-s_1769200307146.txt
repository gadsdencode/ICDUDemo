# TASK 3 — Make AI Judge outputs explainable and actionable (not just scores)
## Requirements
Currently the AI Judge returns IAS/PAS/AS + decision. Upgrade it to also return:
- rationale: bullet list explaining WHY the decision happened
- drivers: list of “score drivers” with references to ICDU fields (e.g., “PAS -20 because principle ‘No PII’ conflicts with input containing email-like string”)
- to_promote: checklist of edits that would move outcome from ESCALATE/BLOCK → PROMOTE

### Deterministic mock scoring (simple heuristics)
Keep it deterministic and local. Suggested heuristics:
- IAS (Intent Alignment):
  - + if intent is specific (contains measurable keywords like “reduce”, “increase”, “by”, “within”, “deliver”)
  - - if intent too vague (short length, generic like “help”, “improve” without target)
- PAS (Principle Adherence):
  - - if input contains “email”, “ssn”, “password”, or looks like PII (basic regex)
  - - if principles include “no legal advice” but intent asks for legal counsel
  - - if principles include “no medical advice” but intent asks for diagnosis
- AS (Application Score):
  - + if persona/tone is specified and consistent
  - - if context/constraints missing or empty
  - - if channel requires brevity (SMS) but tone is “long-form detailed” (if your data model includes channel)

### Gates and decisions
- Show thresholds clearly (e.g., PROMOTE if IAS>=80 AND PAS>=85 AND AS>=75)
- Enforce “principles are non-negotiable”: if PAS < threshold, decision should be BLOCK even if IAS is high.
- Decisions: PROMOTE / ESCALATE / BLOCK

### UI upgrades
- Present scores + thresholds + decision
- Add an “Explain this decision” expandable section:
  - Rationale bullets
  - Drivers list
  - “To promote” checklist
- Add a “View as report JSON” toggle

Acceptance:
- A user can understand *why* the gate blocked and what to change next.
